# 🔥 แนวทางหลัก — Feature เปิดใช้ **ต่อ tenant** ไม่ใช่ flag ในโค้ด

> **นี่คือ priority หลักของโปรเจค** — งานทุกอย่างหลังจากนี้ต้องสอดคล้องกับข้อนี้
> รายละเอียดการตัดสินใจ + ทางเลือกที่ไม่เลือก: [[adr-0007-phase-6-entitlements]]

## กฎข้อเดียวที่ต้องจำ

**ฟีเจอร์เปิด/ปิด = ข้อมูลต่อ workspace ใน DB — เปลี่ยนได้ตอน runtime โดยไม่ต้อง deploy ใหม่**

ห้ามใช้กลไกพวกนี้ในการกั้นฟีเจอร์ที่ขาย:

| ❌ ห้าม                          | ทำไม                                                                                          |
| -------------------------------- | --------------------------------------------------------------------------------------------- |
| `if (process.env.FEATURE_X)`     | เป็นค่าระดับ deployment — ลูกค้าทุกคนบน instance เดียวกันได้/ไม่ได้เหมือนกัน หมด multi-tenant |
| ค่าคงที่ในโค้ด / build-time flag | ลูกค้าจ่ายเพิ่ม = ต้อง build+deploy ใหม่ · ต้อง build ต่อลูกค้า                               |
| fork/branch ต่อลูกค้า            | merge hell ตั้งแต่ลูกค้าคนที่ 2                                                               |
| ซ่อนปุ่มใน UI อย่างเดียว         | ไม่ใช่ security — ยิง API ตรงได้                                                              |

✅ **ที่ถูก:** row ใน `workspace_entitlements` → `hasEntitlement(entitlements, 'reports')` → เช็คที่ **domain/route ฝั่ง server** · ลูกค้าจ่ายเพิ่ม = update 1 row เห็นผลทันที

> ทุน pattern นี้มีมาตั้งแต่ Phase 5 แล้ว: `workspace_bot_config` (`botEnabled`/`aiEnabled` ต่อ workspace ·
> เช็คที่ `bot-consumer.ts` ฝั่ง server · **ไม่มี row = ปิด**) — Phase 6 คือการขยายให้เป็นระบบเต็ม

## 3 หลักที่ตามมา

### 1. สร้าง "กลไก" ก่อน · เขียน "ฟีเจอร์" ตอนมีคนจ่าย

**ห้ามไล่เขียนฟีเจอร์ให้ครบตามใบเสนอราคาก่อนขาย** — 72 ฟีเจอร์ = หลายเดือนถึงเป็นปีสำหรับ solo+AI ·
ฟีเจอร์ที่ไม่มีใครซื้อ = หนี้ maintenance ตลอดชีพ · ใบเสนอราคา (`apps/billing`) มีไว้ **ให้ลูกค้าบอกเรา
ว่าต้องการอะไร** ถ้าเดาเองก่อนก็เสียของ

### 2. ไม่ใช่ทุกอย่างในใบเสนอราคาต้องมี entitlement

| ประเภท                                                       | กลไก                                                                                               |
| ------------------------------------------------------------ | -------------------------------------------------------------------------------------------------- |
| **ช่องทางแชท** (~10)                                         | gate ด้วย **ข้อมูล** — ไม่ได้ซื้อ = ไม่มี row ใน `channels` + ไม่มี credential (มีตั้งแต่ Phase 4) |
| **บริการ** (training, ย้ายข้อมูล, ดูแลระบบ, self-host) (~10) | ไม่ใช่โค้ด — งานคน ไม่มี flag                                                                      |
| **ฟีเจอร์ในระบบ** (~45)                                      | ✅ entitlement ระดับ **โมดูล** (8 ตัว) ไม่ใช่รายฟีเจอร์                                            |

### 3. fail-closed เสมอ

ไม่มี row / ไม่รู้จักโมดูล = **ไม่มีสิทธิ์** — ลืมตั้งค่าแล้วปิด ดีกว่าลืมแล้วเปิดฟรี

## ข้อจำกัดที่ยอมรับแล้ว

- **self-host กันไม่ได้จริง** — ลูกค้าถือโค้ด flip ได้เอง · ตอนนี้ขาย self-host เป็น **trust + support**
  ไม่ใช่ technical lock (ถ้าจะกันจริงต้อง license key + signed entitlement — ยังไม่ทำ)
- **โมดูลหยาบกว่าใบเสนอราคา** — ซื้อ "รายงาน SLA" ตัวเดียวจะได้รายงานทั้งโมดูล · ยอมรับ เพราะไม่มีต้นทุนเพิ่ม
  และแตกละเอียดทีหลังได้แบบ additive · **ความละเอียดในใบเสนอราคาคือเครื่องมือตั้งราคา ไม่ใช่เครื่องมือบังคับสิทธิ์**
